# Eternal Hunger Architecture Audit

## 목적

이 문서는 시뮬레이션 로직, UI/UX, 어드민 편의성, 밸런스 조정 작업을 같은 기준으로 진행하기 위한 현재 구조 점검표입니다. 앞으로 큰 변경을 할 때는 먼저 이 문서의 기준으로 회귀 위험을 확인합니다.

## 현재 핵심 진단

### 1. 시뮬레이션 로직

- `client/src/app/simulation/page.js`가 시뮬레이션 실행, React 상태, 로그, 개발자 도구, 결과 저장, UI 렌더링을 동시에 담당합니다.
- `runEvents`는 구조화 이벤트의 중심이지만, 화면 로그 문자열과 아직 병행됩니다.
- 1초 단위 진행, 배속, 개발자 도구 표시 여부, 결과 요약 계산이 같은 컴포넌트 생명주기에 묶여 있어 성능과 안정성에 영향을 줍니다.

개선 원칙:

- `runEvents`를 원본 데이터로 두고, 화면 로그/MD/JSON 내보내기/결과 요약/밸런스 진단은 파생 데이터로 만듭니다.
- React 컴포넌트는 렌더링과 입력만 담당하고, 시간 진행과 행동 결정은 `_lib` 모듈로 계속 분리합니다.
- 고정 seed와 동일 설정으로 반복 실행했을 때 장비 완성률, 사망 분포, 레벨/숙련도, 오브젝트 획득률이 비교 가능해야 합니다.

### 2. UI/UX

- 모바일 화면은 기능을 모두 노출하려다 정보 밀도가 높아졌습니다.
- 시뮬레이션 페이지는 상단 상태, 중앙 정보, 하단 컨트롤의 우선순위가 더 명확해야 합니다.
- 개발자 도구는 디버그용 대량 데이터와 요약 계산을 포함하므로, 필요할 때만 렌더링해야 합니다.

개선 원칙:

- 모바일은 `상단 상태 / 중앙 탭 / 하단 컨트롤` 구조를 우선합니다.
- 로그, 캐릭터, 맵, 개발자 도구는 동시에 무겁게 렌더링하지 않습니다.
- 초보자에게 낯선 용어는 별도 설명 페이지보다 문맥형 도움말과 요약 문구로 보완합니다.

### 3. 어드민/데이터 관리

- 아이템 트리는 계정별 데이터와 기본 데이터가 공존합니다.
- 대량 생성, 덮어쓰기, 중복 정리, namu 외 삭제는 긴 작업이므로 한 요청으로 끝내기 어렵습니다.
- 현재 배치 처리 기반은 있으나, UI에서 작업 범위와 진행 상태를 더 명확히 보여줘야 합니다.

개선 원칙:

- 어드민 화면에는 항상 `현재 계정 데이터`, `namu 기본 트리`, `시뮬레이션 생성 아이템`, `커스텀 아이템` 수를 보여줍니다.
- 대량 작업은 dry-run, 진행률, 결과 요약, 재시도 가능한 배치 모델을 기본으로 합니다.
- 서버는 계정별 스코프를 유지하고, 기본/legacy 데이터가 섞여 보일 때 UI에서 명시합니다.

### 4. 밸런스

- 체감 조정은 반드시 진단 지표로 확인합니다.
- 우선 추적할 지표는 다음과 같습니다.
  - 1일차 낮 영웅 장비 5부위 완성률
  - 2~4일차 사망 비율
  - 서든데스 진입 전 생존 팀 수
  - 도주 성공/실패/추격 성공 비율
  - 전설/초월 장비 획득 시점
  - 오브젝트 및 차원의 틈 보상 획득 빈도

## 진행 로드맵

### Phase 1. 진단 기반

- `runEvents` 기반 밸런스 진단 모듈을 추가합니다.
- 내보낸 JSON 로그를 CLI에서 분석할 수 있게 합니다.
- 고정 seed 목록과 비교 기준을 문서화합니다.

### Phase 2. 시뮬레이션 분리

- `simulation/page.js`에서 계산/요약/진단 기능을 `_lib`로 계속 이전합니다.
- 개발자 도구가 닫혀 있을 때 대량 요약 계산과 ping 계산을 최소화합니다.

### Phase 3. UI/UX

- 모바일 레이아웃을 컨트롤 중심으로 재정리합니다.
- 로그 영역과 컨트롤 영역이 서로 밀어내지 않도록 높이와 스크롤 책임을 분리합니다.

### Phase 4. 어드민 편의성

- 아이템 관리 화면에 계정 스코프와 데이터 종류 요약을 표시합니다.
- 대량 작업 메시지를 작업 단위와 진행률 중심으로 정리합니다.

### Phase 5. 밸런스 튜닝

- 룰셋에 진단 기준값을 추가합니다.
- 중반 교전/사망률, 도주 과다, 서든데스 간보기 문제를 룰셋 값으로 조정합니다.
